Advanced BI — Semantic Model & Row-Level Security

Built a reusable semantic model: calculation groups for time intelligence, role-playing dates (Order/Ship), and RLS by Region & Manager. Documented lineage and measures catalogue.

  • Model: Composite model; shared date
  • DAX: Calc group for MTD/QTD/YTD
  • Security: RLS by Region role

Advanced BI dashboard showcasing semantic model behavior, time-intelligence patterns via calculation groups, and region-based role-level security views.

Business Problem

Multiple teams needed consistent KPIs across reports (sales, finance, operations), but each workbook was defining calculations slightly differently. There was also a need to restrict data access based on region and management hierarchy without creating separate reports for each audience.

Data

  • Fact tables for orders and shipments, each with its own date fields and related dimensions.
  • Shared dimensions for Customers, Products, and a central Date table.
  • User attributes such as Region and Manager used to drive row-level security rules.

Solution

  • Designed a reusable semantic model with a single Date dimension acting as a role-playing date for both Order and Ship dates.
  • Implemented a calculation group to standardize time-intelligence patterns (MTD, QTD, YTD, rolling periods, YoY comparisons).
  • Defined Row-Level Security (RLS) roles that filter data by Region and Manager, ensuring users see only their relevant slice of data.
  • Documented a measures catalogue and table-level descriptions so future reports can plug into the same model.

Key Insights

  • Semantic consistency eliminated differences in how core KPIs (revenue, margin, volume) were calculated across reports.
  • RLS allowed a single report to serve many audiences – executives, regional managers, and analysts – without duplicating logic.
  • Calculation groups reduced the number of DAX measures and made it easier to add new time-based views.

Deliverables

  • PBIX model file: Semantic model with tables, relationships, and calculation group.
  • RLS roles: Defined security roles by Region and Manager for controlled access.
  • Documentation: Measures catalogue, model diagram, and RLS design notes.

If This Were in Production

  • Host the model in a centralized workspace as a shared dataset for multiple reports.
  • Integrate with Azure AD security groups for automated user-to-role assignment.
  • Version and test changes to measures and roles before promoting to production.
